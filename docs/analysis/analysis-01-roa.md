# 要求分析: リソース指向設計

## 1. 前提とスコープ

- **対象範囲:**
  - ユーザーに、Web API設計におけるRESTfulとは何かを理解してもらうことを目的とした機能を提供する。
  - HTTP/1.1のメソッド（例: GET, POST）の意味までを理解していない利用者を対象とする。
  - リソースの考え方、パス設計、HTTPメソッドとリソース操作の関係に触れる。
    なお、「リソースとは何か」といった業務要件によって解釈がぶれやすい概念については、固定的な定義を提示しない。

- **制約:**
  - プログラミング言語および実行環境は、Java 21 以降、Spring Boot（Servlet / WebMVC）とする。
  - ユーザーは学習目的で、1日（約8時間）程度で実施可能な内容とする。
  - WebAPIのエラーメッセージは、`RFC 9457: Problem Details for HTTP APIs` に従う
  - HTTPを利用したWebAPIを前提とし、`RFC9110: HTTP Semantics` や `RFC9112: HTTP/1.1` に従う

- **想定外:**
  - プログラミング言語やフレームワークの一般的な利用方法を理解してもらうことは、本スコープには含めない。
  - ただし、RESTfulの理解に最低限必要な範囲での補足は、この限りではない。

## 2. 現状調査

- **ユーザー観察 / インタビュー:**
  - これまでサービス指向設計など、別の設計パラダイムで実装を行ってきた技術者が、
    リソース指向設計の一形態であるRESTful Web APIを開発するにあたり、前提知識の整理と理解が必要となっている。

- **既存システムの振る舞い:**
  - 本資料は教育目的であるため、既存システムの制約は受けない。

- **ギャップ整理:**
  - リソース指向による設計ができるようになること。
  - リソース指向「以外」の設計パラダイムでの実装経験と、
    RESTful Web APIにおけるリソース指向設計との差分を整理し、実装に結びつけられるようにする。

## 3. 競合・代替手段

- **比較対象:**
  - 教育資料を目的とするため、明示的な比較対象は設定しない。

- **採否理由:**
  - 原則は前述の通りとする。
  - ただし、将来的にOpenAPI Specificationを用いた設計・開発を行うことを前提とする。

## 4. リスクと仮説

- **リスク一覧:**
  - 既存の設計パラダイムとの混同
    - 話題がどの設計パラダイムに属するものかが不明確となり、リソース指向設計との対応付けができなくなる可能性がある。
    - オープンなWeb APIの利用経験が乏しい、または自組織内に閉じた環境での開発経験のみの場合、
      知識を補完するための外部事例に触れる機会が少ない可能性がある。
    - 比較を行う際には、どの設計パラダイムについての説明なのかを明確に定義・提示する必要がある。

- **検証すべき仮説:**
  - 別の設計パラダイムで実装を行ってきた技術者が、
    RESTful Web API設計を理解・実践できるようになるための成否条件を定義する。
  - 成否条件に基づき、検証計画（実験、PoC、ユーザーテスト等）を策定する。

## 5. 受け入れ基準の確認

- **ビジネス / ユーザー価値:**
  - ユーザーは、リソース指向によるWeb API設計手法であるRESTfulを用いた
    Web APIインターフェース設計に着手するための前提知識を得られる。
  - OpenAPI Specificationを用いた開発を行う際の前提知識を得られる。

- **測定指標:**
  - 提供機能に基づくアンケートを実施し、理解度を測定する。
